# Optical Flow Sensor — контекст проекта (fallback для Claude Code)

## Кто я и что делаю

Разрабатываю кастомный optical flow сенсор для дрона (GPS-denied
positioning) на связке:

- **MCU:** STM32H743VIT6 (WeAct MiniSTM32H743 dev board, куплена на Ozon)
  — Cortex-M7 @ 480 MHz, 2048 KB Flash, 1 MB RAM, 8 MB SPI Flash,
  8 MB QSPI Flash, microSD, USB-C, разъём камеры 24-pin FPC 0.5mm
  под DCMI, 2× FDCAN, 4× I2C, 4× UART, 6× SPI
- **Камера:** ATK-OV7725 MD (ALIENTEK) — сенсор OV7725 (VGA 640×480,
  rolling shutter, до 60fps VGA / 120fps QVGA) + **FIFO буфер AL422B**
  на борту (важно: НЕ прямой DCMI, см. ниже)
- **IMU:** ICM-42688-P (для де-ротации потока по гироскопу)
- **Дальномер:** VL53L1X ToF (для масштабирования flow по высоте)
- **Связь с FC:** CAN bus, протокол **DroneCAN** через **libcanard**,
  трансивер **TJA1051T** (3.3V native, не TJA1050 — тот 5V)
- **Полётный контроллер:** Pixhawk/Cube, прошивки PX4 и ArduPilot
  (оба варианта должны поддерживаться)

## Важный нюанс по камере: FIFO, не прямой DCMI

ATK-OV7725 MD имеет чип AL422B FIFO буфер между сенсором OV7725 и
выходными пинами. Это значит:

```
OV7725 sensor → AL422B FIFO → STM32 pins
```

**НЕЛЬЗЯ подключить напрямую к периферии DCMI** как обычную DVP
камеру — FIFO управляется отдельными сигналами:
- `WRST` — Write Reset (сброс указателя записи)
- `RRST` — Read Reset (сброс указателя чтения)
- `WEN` — Write Enable (от VSYNC сенсора)
- `OE` — Output Enable (активный LOW)
- `RCLK` — Read Clock (тактируется вручную с GPIO или таймером)
- `D0-D7` — данные (8 GPIO на вход)
- `SCL/SDA` — SCCB (аналог I2C) для конфигурации регистров OV7725

Управление SCCB конфигурацией сенсора — отдельно от чтения кадра
из FIFO. Чтение кадра — программный битбэнг RCLK + чтение GPIO порта
(можно ускорить через TIM+DMA вместо чистого битбэнга).

Распиновка модуля (18 пин 2×9, шаг 2.54мм) уже выяснена и обсуждалась
в предыдущем диалоге — нужно её восстановить из документации ALIENTEK
ATK-OV7725 MD V1.1 если понадобится точная схема.

## Архитектура устройства (целевая)

```
ATK-OV7725 MD (FIFO read)
         │ GPIO битбэнг (8-bit data + WRST/RRST/WEN/OE/RCLK)
         │ + I2C/SCCB для конфига сенсора
         ▼
STM32H743VIT6
  ├── Захват кадра из FIFO → grayscale buffer 64x64 (или больше)
  ├── SAD optical flow algorithm (порт PX4Flow)
  ├── Де-ротация через IMU (ICM-42688, SPI)
  ├── Масштабирование по высоте (VL53L1X, I2C, дальномер)
  ├── Упаковка в DroneCAN сообщения (libcanard)
  └── FDCAN → TJA1051T → CAN bus
                              │
                              ▼
                    Pixhawk/Cube (PX4 или ArduPilot)
```

DroneCAN сообщения для публикации:
- `com.hex.equipment.flow.Measurement` (Data Type ID 20000) —
  integrated_x/integrated_y (рад/с), quality
- `uavcan.equipment.range_sensor.Measurement` (ID 1050) — дальномер
- `uavcan.equipment.ahrs.RawIMU` (ID 1003) — опционально, если нужно
  делиться сырым IMU с FC

PX4 параметры: `UAVCAN_ENABLE=3`, `UAVCAN_BITRATE=1000000`,
`EKF2_OF_CTRL=1`, `EKF2_RNG_CTRL=1`

ArduPilot параметры: `CAN_P1_DRIVER=1`, `CAN_D1_PROTOCOL=1`,
`FLOW_TYPE=6`, `RNGFND1_TYPE=24`

### Откуда взят алгоритм

Источник: https://github.com/PX4/PX4-OpticalFlow (современный, активно
поддерживаемый репозиторий PX4, наследник старого PX4Flow). Конкретно
`src/px4flow.cpp` — класс `PX4Flow`, метод `compute_flow()`.

(Также упоминались для справки, но НЕ использовались: PX4/PX4-Flow
(старый репозиторий, NuttX-специфичный) и PX4/snap_cam (Linux camera
streaming, не релевантно для bare-metal STM32))

### Как устроен алгоритм (важно для дальнейшей работы)

1. Изображение делится на сетку `NUM_BLOCKS × NUM_BLOCKS` (5×5) опорных
   блоков `TILE_SIZE` (8×8 пикселей)
2. Для каждого блока `compute_diff()` проверяет "текстурность" (сумма
   построчных + постолбцовых разностей в 4×4 центре блока) — слишком
   гладкие блоки (ниже `flow_feature_threshold=30`) пропускаются
3. Для оставшихся блоков `compute_sad_8x8()` — полный перебор SAD
   (Sum of Absolute Differences) в окне `[-search_size, +search_size]`
   = `[-4, +4]` пикселей во втором кадре
4. Находится позиция минимального SAD → целочисленный сдвиг `(dx, dy)`
5. `compute_subpixel()` — уточнение по 8 направлениям вокруг найденного
   минимума через билинейную интерполяцию соседних пикселей, выбирается
   направление с наименьшим SAD → субпиксельный сдвиг шагом 0.5px
6. Усредняются (ариметически — **не медиана**) все валидные сдвиги
   блоков → итоговый flow в пикселях за кадр
7. `quality = meancount * 255 / (NUM_BLOCKS * NUM_BLOCKS)` — чем больше
   валидных блоков, тем выше quality (0-255)

В оригинальном C++ используются ARM SIMD интринсики (`__USAD8`,
`__USADA8`, `__UHADD8`) для скорости — в Python это эмулируется через
NumPy. **На STM32H743 эти интринсики доступны нативно** как CMSIS DSP
функции (Cortex-M7 имеет аппаратную поддержку SIMD для 8-bit операций).

### ⚠️ КРИТИЧЕСКИ ВАЖНО: соглашение о знаке результата

Алгоритм возвращает смещение **текстуры сцены между кадрами**, а НЕ
смещение камеры. Если камера физически сдвинулась вправо на `+N`
пикселей, текстура в кадре сдвинулась влево — `compute_flow` вернёт
`flow_x = -N`.

Это подтверждено на синтетическом тесте: при `true_vx=2.0, true_vy=1.0`
алгоритм стабильно возвращал `flow=(-2.00, -1.00)` с нулевым
разбросом (`std=0.0000`).

Это оригинальное поведение `PX4Flow::compute_flow` и именно так PX4
EKF2 ожидает данные в DroneCAN-поле `integrated_x`/`integrated_y`.

**При портировании на STM32 для реальной публикации в DroneCAN знак
НЕ инвертировать.** В Python-прототипе есть параметр
`invert_sign=True` в `compute_flow_fast()` — это ТОЛЬКО для удобной
визуализации стрелки на экране (используется в `flow_camera_test.py`
для наглядности), на железе использовать оригинальный знак.

### Производительность (замерено)

Чистый Python на блоке 64×64: ~3.5-5 мс/кадр (векторизованная
`compute_flow_fast`), что эквивалентно ~250 fps. Это нижняя оценка
снизу для C-кода на STM32H743 — на железе с CMSIS DSP интринсиками
ожидается **1-3 мс/кадр** (480 МГц Cortex-M7 + аппаратный SIMD), то
есть теоретически 300-1000+ fps, реально лимитируется частотой кадров
с камеры (60fps VGA / 120fps QVGA для OV7725).

## Обсуждались, но НЕ реализованы — улучшения алгоритма

Список идей по приоритету (от пользователя был запрос "как улучшить
алгоритм, не оптимизацию кода, а именно улучшить" — ответ дан, но
код НЕ писался, только план):

### Приоритет 1 — дёшево и сразу полезно

1. **Медиана вместо среднего.** В оригинальном коде гистограммный
   фильтр пиков **фактически отключён** — `if (1) {...}` вместо
   `if (histx[maxpositionx] > meancount/6 && ...)`. Это похоже на
   баг/недоделку оригинала. Один плохой блок (тень, блик, движущийся
   объект) тянет среднее наравне с хорошими. Исправление: взять
   честную медиану по `dirs_x`/`dirs_y` вместо арифметического
   среднего, либо восстановить и реально использовать гистограммный
   пик для отбраковки выбросов.

2. **Forward-backward consistency check.** Посчитать flow от
   img1→img2, затем от img2→img1 для того же блока, проверить что
   они примерно противоположны (`flow_fwd ≈ -flow_bwd`). Если нет —
   блок ненадёжен, выбросить. Недорого (можно делать только для
   подмножества блоков), сильно режет outlier-ы.

3. **Уникальность минимума SAD (anti-aperture problem).** Сейчас
   берётся просто минимальный SAD в окне поиска. Проблема: на
   однородных повторяющихся текстурах (трава, асфальт) может быть
   несколько похожих минимумов — алгоритм выбирает один из них
   почти случайно (aperture problem). Улучшение: проверять не только
   абсолютное значение SAD в найденном минимуме, но и разницу с
   следующим по близости локальным минимумом. Если разница маленькая
   — блок ненадёжен, отбросить. Почти бесплатно по вычислениям (одно
   дополнительное сравнение).

4. **Параболическая субпиксельная интерполяция.** Текущий метод
   (`compute_subpixel`) — грубое билинейное усреднение по 8
   направлениям, шаг квантования 0.5px. Лучше: параболическая
   интерполяция по трём точкам SAD (минимум + два соседа по каждой
   оси отдельно по X и Y) — даёт непрерывную субпиксельную оценку,
   точность растёт в разы, вычислительно дешевле текущего метода
   (не нужно для каждого пикселя блока считать 8 средних — только
   3 SAD-значения по каждой оси).

### Приоритет 2 — дороже, но даёт прирост диапазона/устойчивости

5. **Image pyramid (coarse-to-fine).** Текущий `search_size=4` —
   жёсткий потолок максимально детектируемой скорости (текстура не
   может сместиться больше чем на 4px между кадрами, иначе алгоритм
   не найдёт верный минимум). Пирамида: грубый поиск на уменьшенном
   вдвое/вчетверо изображении находит примерный сдвиг, затем
   уточнение на полном разрешении локально вокруг этого сдвига.
   Снимает ограничение диапазона скорости почти бесплатно по
   вычислениям относительно roста окна поиска напрямую.

6. **Census transform или NCC вместо SAD.** SAD чувствителен к
   глобальным изменениям яркости между кадрами (автоэкспозиция
   камеры подстраивается постоянно на OV7725 — это бьёт по точности).
   Census transform особенно дёшев (битовые операции), устойчив к
   изменению яркости/контраста.

### Приоритет 3 — специфично под rolling shutter камеру (OV7725)

7. **Rolling shutter компенсация.** OV7725 — rolling shutter, не
   global shutter (это известное ограничение, обсуждалось при выборе
   камеры — MT9V034 был бы global shutter, но не найден в продаже,
   решили на прототип взять ATK-OV7725 MD с компенсацией через
   гироскоп). При повороте дрона во время экспонирования кадра верх
   и низ изображения сняты в разные моменты — систематическое
   искажение flow на высоких угловых скоростях. Компенсация: зная
   угловую скорость с гироскопа и время чтения строки матрицы, можно
   скорректировать построчно, либо хотя бы детектировать "подозрительные"
   кадры (резкий поворот по гироскопу) и понижать quality на них.

8. **Учёт motion blur.** При резких движениях SAD-метрика деградирует
   предсказуемым образом — можно детектировать высокий общий SAD по
   всем блокам сразу и понижать quality, сигнализируя FC что данным
   flow доверять не стоит в этом кадре.

### Приоритет рекомендован для железа пользователя (STM32H743 +
rolling shutter, ограниченный вычислительный бюджет на старте)

Порядок реализации: 1 (медиана) → 2 (forward-backward) → 4
(параболическая интерполяция) → 3 (anti-aperture) → затем по
возможности 5/6/7/8.
