## Выполнимо сейчас (софт + имеющиеся данные)

Приоритет по ценности×простоте:

1. **Сравнение с DIS-Flow и LK** (OpenCV `DISOpticalFlow`, `calcOpticalFlowPyrLK`).
   Прогнать на баге, сравнить траектории/resid. Позиционирует наш лёгкий блок-матчер
   против стандартов. Быстро, информативно. → **тастер сделан, см. results/bag_vs_optflow.png**

2. **RANSAC для global motion**: вместо медианы фитить глобальную трансляцию по блокам через
   RANSAC. → **ПРОВЕДЕНО, отклонён.** Шум −57% (0.56→0.24, находит плотный консенсус), НО
   яркость +38% (1.5→2.1): при дрейфе gain блоки разъезжаются, узкий tau цепляет неверный
   кластер. MAD даёт бóльшую часть выигрыша по шуму без поломки яркости. clean/outlier ≈.

3. **Пространственная карта надёжности (spatial reliability map)**: → **ПРОВЕДЕНО**
   (`results/spatial_reliability.png`). Гипотеза «края хуже» подтверждена ЧАСТИЧНО, механизм
   ДРУГОЙ: плохая только верхняя полоса кадра (валидных 0.68 vs 0.90), симметричные края ок.
   Острота пика по краям НЕ хуже (431 центр vs 441 края) → **виньетирования/дисторсии нет**.
   Причина — конструкция дрона в верху кадра (нога/луч ловит не землю). Фикс: маска верхней
   зоны под КОНКРЕТНЫЙ монтаж, а не универсальное центр-взвешивание. Actionable, но
   mount-specific.

4. **Temporal filtering** (EMA): → **ПРОВЕДЕНО, отклонён.** Чуть чистит путь
   (2.01→1.76 при α=0.3), но точность НЕ растёт (resid 1.06→1.09, лаг размывает движение).
   Путь остаётся ~2× даже при сильном сглаживании → инфляция это низкочастотный дрейф
   зависания, не джиттер. EKF сам фильтр — префильтрация избыточна и вредна (лаг).

5. **Temporal drift analysis**: → **ПРОВЕДЕНО** (`results/temporal_drift.png`). Ключ: **дрейф
   НЕ накапливается** за 155 с open-loop — скорость роста ошибки ≈0 (зависание −0.002 м/с,
   движение −0.004 м/с). Ошибка ограничена ~1–3 м, финальная 0.72 < средней 0.90. Разрешает
   загадку «путь 2×»: это локальные петли, которые уходят и ВОЗВРАЩАЮТСЯ, а не накопление.
   Позиция точна несмотря на «грязный» путь → 2× косметика, не проблема точности.

6. **Размер блока (TILE_SIZE)**: → **ПРОВЕДЕНО, условно-положительный.** T=16 бьёт шум −36%
   (0.56→0.36) и яркость −47% (1.5→0.79) — больше пикселей усредняется. clean/outlier ≈. НО
   цена TILE²: T=16 = 4× SAD, T=12 = 2.25× (горячий цикл STM32). Реальная ручка
   точность/вычисления: T=8 дефолт по бюджету, T=12/16 если есть запас. Для адаптива — под
   железо. (Требует параметризации хардкоженных 8 в SAD-функциях, если внедрять.)

7. **Weighted SAD**: → **ПРОВЕДЕНО, отклонён.** Гауссов центр-вес хуже везде (теряет инфу
   с краёв блока). Градиент-вес помогает только яркости (−42%), но портит clean/noise/outlier
   — меняет одно на многое. Равномерный SAD правильнее; яркость лучше через census/zero-mean.

8. **Self-supervised калибровка FOV**: → **ПРОВЕДЕНО, ограничено (подтвердилось).** resid
   плоский (1.058) при любом focal — Procrustes scale-инвариантен, по форме не калибруется.
   По масштабу scale=1 при focal≈118px (FOV 44°) vs паспортный 110px (47°), ~7%. Но эталон
   сам OF-ведомый → это подгонка нашего OF к ИХ OF, не к реальности. Для абсолютного FOV
   нужен независимый RTK-GPS (в баге нет). Работает как относительная сверка, не абсолютная.

## Математические метрики совпадения (`results/match_metrics.png`)

- **Фазовая корреляция (FFT)**: → **ПРОВЕДЕНО, не для per-block.** Глобальная (весь кадр)
  сильна на шуме (0.56→0.23) и яркости (1.5→0.28) — инвариантна к яркости (фаза) и
  усредняет в частотной области. НО outlier катастрофа (2.2) — глобальная оценка не
  отбраковывает выбросы. Блочная (32×32) не взлетела (мало блоков). Как RANSAC: хороша на
  шуме/яркости, слепа к выбросам. + FFT-стоимость.
- **ZNCC (zero-mean normalized cross-corr)**: → **ПРОВЕДЕНО, ниша.** Полная инвариантность к
  offset+gain. **Лучший на clean (0.208→0.164, −21%!), bright (−89%), outlier (−18%)** —
  золотой стандарт метрики, точнее на чистом сигнале. НО шум +85% (нормировка на std раздувает
  шум в низкоконтрастных блоках), на bright+шум high-pass бьёт (0.50 vs 1.35). Дороже (умнож.+
  норм.). Вывод: ZNCC — если мало шума и есть вычисления; high-pass — для реального шума.
- Общий вывод: все яркостно-инвариантные методы (census, high-pass, ZNCC, фаза) фундаментально
  усиливают шум (убирают НЧ → ВЧ-шум растёт). Бесплатной яркостной инвариантности нет; лучший
  путь — фикс экспозиции в камере.
- **Тензор структуры (Shi-Tomasi λ_min)**: → **ПРОВЕДЕНО, нишевый детектор aperture.** λ_min из
  градиентов img1 (до матчинга) = насколько поток определён. Ловит **полосы/грани** (stripes
  q=100→0 при λ>500), но НЕ плитку (периодика: градиенты в обе стороны → высокий λ_min).
  Вскрыло: два разных механизма aperture — **грани ловит тензор структуры, периодику ловит
  uniqueness (peak-ratio)**, комплементарны. На нормальных сценах точность не меняет (блоки уже
  фильтрованы по текстурности). НЕ провалился как взвешивание по кривизне (меряет текстуру, не
  совпадение → окклюдер не обманет). Опция для сцен с линейными структурами (дороги/борозды).

## Прочие проведённые

- **Pixel-locking коррекция субпикселя** (`use_subpix_correction`): → **ПРОВЕДЕНО, ВАЖНЫЙ
  УРОК.** Из литературы: SAD+парабола систематически тянет субпиксель к целым (bias 0.13px,
  истина 0.3→оценка 0.1, V-образная SAD не подходит параболе). Замерил кривую, построил
  корректор (LUT-инверсия). **Синтетика: clean −72%, outlier −70%** (субпиксельный bias был
  доминирующей ошибкой в чистых условиях!). НО **на реальном баге ХУЖЕ: resid 1.27→1.64
  (+29%)**. Причина: LUT калиброван на scipy bilinear-сдвиге = **переобучение на артефакт
  тест-генератора**; реальная камера имеет другую субпиксельную статистику + шум/смаз, и
  синтетический корректор усиливает ошибки. Методологический вывод: **синтетический выигрыш
  ОБЯЗАН проверяться на реальных данных** — иначе это фитинг под генератор. Флаг оставлен для
  ре-калибровки LUT на реальной камере (техника реальна, кривая — нет). SSD вместо SAD даёт
  меньший bias (0.104 vs 0.129) без усиления шума, но дороже.

- **Illumination pre-normalization (high-pass)**: → **ПРОВЕДЕНО, ПРИНЯТО как опция**
  (`use_highpass`). Вычесть локальное среднее (бокс-фильтр) перед SAD -> яркостная
  инвариантность. Тот же фундаментальный размен что census (яркость↓ ценой шума↑, т.к. НЧ
  убирается -> ВЧ-шум усиливается), НО мягкое ядро (k=13) даёт яркость −86% при штрафе шума
  всего +14% (census +83%). Работает на реальном баге (resid 1.06→1.05, scale 0.937→0.949,
  valid 1641), где census ломался (66 валидных). Дешевле (сепарабельный бокс). **Заменяет
  census** как основная опция для автоэкспозиции. LCN (нормировка контраста) — хуже (шум ×2).

## Мульти-сенсор (избыточность)

- **Мульти-IMU для де-ротации (синт, реальная гиро-трасса)**: → **ПРОВЕДЕНО**
  (`results/multi_imu.png`). Гироскоп используется для де-ротации (вычесть `f·ω·dt`); качество
  упирается в шум/биас/сбои гиро. Monte-Carlo 300 реализаций, ICM-42688 уровень (шум 0.06°/с,
  биас между IMU 0.15°/с):
  - **√N снижение**: дрейф де-ротации 1 IMU=5.9м → 2 IMU=4.3 → 3=3.4 → 4=2.8м (строго 1/√N).
    4 IMU почти вдвое режут дрейф.
  - **Отказ критичен**: 1 сбойная IMU из 3 при наивном СРЕДНЕМ = 20м дрейфа (ХУЖЕ одной
    исправной!). МЕДИАНА-голосование отбрасывает сбой → 5.2м. Вывод: несколько IMU + робастная
    медиана, НЕ среднее. 3 IMU = минимум для голосования (выкинуть 1 сбой).
  - Честно: абсолютный дрейф bias-доминирован, реальный EKF оценивает биас гиро онлайн →
    меньше. Но √N и отказоустойчивость реальны. Common-mode вибрацию усреднение НЕ убирает
    (все IMU на жёстком монтаже видят её одинаково) — нужна мех. развязка + фильтр.
  - IMU дёшевы/крошечные (ICM-42688 ~$), низкий счёт → 3 IMU = дешёвый выигрыш надёжности.

## Нужны доп. данные / железо

- **Multi-camera fusion (видимый + ИК)**: → **ПРОВЕДЕНО, ПОЛОЖИТЕЛЬНО**
  (`results/multicam_fusion.png`). `/cam_usb` = видимый (= `/odometry/image`, VO-камера),
  `/cam_usb0` = ИК, обе вниз, перекрывающийся обзор, одинаковые интринсики. Прогон нашего
  потока на обеих + взвешивание по quality: vis 1.33 м, ИК 2.05 м (хуже — меньше текстуры),
  **fused 1.19 м (−11% vs vis)** + чуть выше availability. ИК душится где плох, но добавляет
  комплементарный сигнал где видимая слаба. Кэш `data/bag_flight/multicam.npz`.
  Дальше: fore-aft база даёт скорость по временной задержке (не пробовано).

- **Массив 3-4 downward-камеры (синт)**: → **ПРОВЕДЕНО** (`results/array_yaw.png`). Массив
  разнесённых камер восстанавливает **yaw БЕЗ гироскопа** через аффинный фит поля потока
  `f_k = t + M·r_k` (differencing изолирует вращение от трансляции). Результаты:
  - **4 на лучах** (база 0.35м, h=2м): ошибка yaw **4°** — точно (45→45, 90→89, 150→151).
  - **3 треугольник** (база 0.1м): ошибка **11°** — занижает, малая база = слабый сигнал.
  - **высота бьёт**: 4-луча на h=8м → ошибка 13° (сигнал ∝ b/h, на высоте тонет в шуме).
  - трансляция восстанавливается точно (√N усреднение): v=(1.0,-0.5)→(0.94,-0.49).
  ЧЕСТНО: массив даёт только **yaw** (curl поля). Pitch/roll — равномерный сдвиг образа,
  сокращается в differencing → неотличим от трансляции, нужен гиро или омни. И yaw слабеет
  на высоте. Вывод: массив = робастность/избыточность (√N) + yaw-подмога/дубль гиро на
  малой высоте с БОЛЬШОЙ базой (лучи>>треугольник), НЕ полная замена IMU. Лучшая конфиг для
  позиционирования: 4 на лучах + IMU (гиро для pitch/roll и высоты, массив для yaw+шума).
- **Self-supervised cam-IMU offset + temporal delay**: минимизация ошибки EKF2. Нужен
  настоящий независимый эталон (RTK-GPS) — в текущем баге его нет (OF инжектится как GPS).
- **Rolling shutter компенсация** (приоритет-3 плана): нужен реальный RS-сенсор + синхро-IMU.
