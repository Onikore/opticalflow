# Optical Flow сенсор — улучшенный PX4Flow + стенды

Python-порт и **улучшенная версия** алгоритма SAD block-matching из
[PX4/PX4-OpticalFlow](https://github.com/PX4/PX4-OpticalFlow) для кастомного
optical-flow сенсора GPS-denied позиционирования дрона (цель — **STM32H743 +
ATK-OV7725 MD**). Тестовый стенд на ПК: синтетика, реальное видео, реальный
полёт из rosbag, стенды де-ротации / DroneCAN / рабочего диапазона / безопасности.

Каждое улучшение — отдельный флаг в `compute_flow_improved`; при всех OFF
результат **бит-в-бит совпадает с оригиналом PX4Flow** (проверено на 800 парах).

## Структура репозитория

```
src/       ЯДРО optical flow: алгоритм + стенды. Запускать из корня: python3 src/xxx.py
nav/       GNSS-denied НАВИГАЦИЯ (отдельно от ядра): мозаика/спутник/локализация/баги
c/         референсная C-реализация (px4flow_ref.c)
data/      videos/ (снятые .MOV), real_frames/ (текстуры .npy),
           bag_flight/ (кэши реальных полётов), test_frames/ (golden для C-порта)
results/   flow/ — результаты ядра (scoreboard, сравнения алгоритмов, сенсорика)
           nav/  — результаты навигации (траектории полётов, мозаики, спутник)
docs/      FINDINGS.md, EXPERIMENTS.md, fallback.md (контекст проекта)
```

Кэши полётов: `bag1_cam_cache.npz` / `bag2_cam_cache.npz` — **чистая камера
`/cam_usb/image/compressed`** (105 Гц → дедуп + каждый 8-й = 12.5 Гц). Топик
`/odometry/image` — ДЕБАГ с нарисованными маркерами фич, для замеров непригоден
(см. аудит в `docs/EXPERIMENTS.md`).

Пути к данным/выходам считаются от корня репо (`src/_paths.py`) — скрипты
работают из любого каталога.

**Алгоритм** (`src/`): `px4flow_algo.py` — эталон 1:1 как C++ (референс C-порта);
`px4flow_fast.py` — векторизованная (идентична); `px4flow_improved.py` —
улучшенный, флаги + self-check (`python3 src/px4flow_improved.py`).

**Интеграция** (`src/`): `flow_to_velocity.py` — поток → скорость центра дрона
(де-ротация по гиро + lever-arm смещения камеры `−ω×r` + flow→м/с + DroneCAN-поля).
Коррекции, что не входят в блок-матчинг, но нужны на выходе сенсора для EKF.

**Стенды** (`src/`): `benchmark.py` (RMSE по сценариям), `test_equivalence.py`,
`synthetic_test.py`, `video_test.py` (реал.текстура + Farneback), `mov_test.py`
(видео→метры), `algo_compare.py` (наш vs DIS vs LK на полёте), `derotate_test.py`,
`dronecan_test.py`, `quality_test.py`, `envelope_test.py`, `make_panel.py`,
`flow_camera_test.py`, `golden.py` (генерация/приёмка golden-набора).

**Навигация без ГНСС** (`nav/`): `rosbag_path.py` — полёт из mcap-бага → кэш →
траектория/видео/гейты; `map_localize.py` — абсолютная локализация по геопривязанной
подложке (своя мозаика или спутниковый снимок через `from_image`) → fix →
`enu_to_wgs84` → GPS_INPUT ⇒ миссии QGC/MP в WGS84 без ГНСС; `fetch_sat.py` —
скачивание спутниковых тайлов; `sat_test.py` — кадры бага против спутника.
Демо `map_localize` (teach-repeat, чистая камера): 65% фиксов, медиана 0.42 м,
p90 1.03 м.

## Алгоритм улучшения и флаги

Улучшения тестировались итеративно через `benchmark.py` (RMSE в px, меньше =
лучше) на 4 сценариях: субпиксель / шум σ=6 / дрейф яркости (gain 1.12, автоэкспозиция
OV7725) / выброс-окклюдер — и валидировались на реальном полёте (rosbag).

### Принятый дефолт

**median + parabolic + tri_subpix + pyramid + boundary_reject + zsad_fine +
`flow_feature_threshold=15`** (scoreboard на реальной текстуре,
`results/flow/scoreboard.png`): clean **−57%**, шум **−63%**, яркость **−80%**, выбросы
**−87%** vs оригинал PX4Flow, без регрессий. На реальном полёте (чистая камера
`/cam_usb`, 109 м): **resid 2.76 → 1.23 м** (`results/flow/bag_combos.png`).
MAD после аудита на чистой камере — **условный флаг, не дефолт** (на слаботекстурной
земле вредит: 1.23→1.30).

| Улучшение | Флаг | Эффект |
|---|---|---|
| Медиана вместо среднего | `use_median` | шум −61%, яркость −35%, выбросы −74%. Правильно выкидывает «застрявшие» блоки (смаз/rolling shutter), которые тянут среднее |
| Параболический субпиксель | `use_parabolic` | clean −48%, дешевле оригинального 8-направленного метода (3 значения SAD по оси) |
| Триангулярный субпиксель | `use_tri_subpix` | V-образная интерполяция (правильна для L1/SAD, парабола предполагает U-форму) → pixel-locking bias −39% (0.144→0.088 px). clean −38%, выбросы −36%, **честна без LUT** (в отличие от `use_subpix_correction`, что переобучился) и не вредит на реале. Рекомендуется поверх параболы |
| Пирамида (coarse-to-fine) | `use_pyramid` | снимает потолок `search_size=4` (быстрые 5–9 px/кадр −99%). Грубый поиск на ½-разрешении на **zero-mean SAD** (яркостно-устойчив — иначе ломался на дрейфе яркости) |
| Отказ по краю окна | `use_boundary_reject` | минимум SAD на границе `±search_size` → блок ненадёжен → quality 255→~140 вместо confident-garbage, без гироскопа. Zero harm на нормальном движении |
| MAD-фильтр медианы | `use_mad` | ⚠️ УСЛОВНЫЙ после аудита: на синтетике шум −17%, яркость −26%, но на чистой камере над слаботекстурной землёй ВРЕДИТ (resid 1.23→1.30, k=2–4 не спасает). Включать только над контрастной поверхностью |

На реальном полёте (чистая камера) наш блок-матчер **точнее LK** (1.23 vs 2.23 м)
и уступает DIS-Flow ~28% (0.88 м) — при том, что DIS на STM32 не портируется, а наш
целочисленный тракт — да (`results/flow/bag_vs_optflow.png`,
`results/flow/algo_trajectories.png`).

### Опциональные флаги

Под конкретные условия, не в дефолте (детали и цифры — в `docs/`):

| Улучшение | Флаг | Когда / почему не дефолт |
|---|---|---|
| High-pass (яркость) | `use_highpass` | вычесть локальное среднее перед SAD → инвариантность к яркости. **Предпочтительнее census**: та же польза (−86%), но штраф на шум вдвое меньше (+14% vs +83%), дешевле, работает на реальном баге |
| Census transform | `use_census` | Hamming по бинарным подписям, инвариант к яркости. Хуже high-pass (шум +83%, ломается на реальной съёмке). Запасной |
| Уникальность (anti-aperture) | `use_uniqueness` | peak-ratio: ловит confident-garbage на повторяющейся текстуре (полосы/плитка). Цена — availability. Над плиткой/полем + фикс.экспозиция |
| Motion-blur quality | `use_peak_quality` | понижает quality по кривизне корр. пика. Уточнение (базовый quality уже реагирует). NB: «высокий общий SAD» блюр НЕ ловит (измерено) |
| Forward-backward | `use_fb_check` | +2× счёта, рушит availability при дрейфе яркости. Плохой размен |
| Субпиксель pixel-locking | `use_subpix_correction` | ⚠️ синтетика −70%, но на **реальном баге ХУЖЕ +29%** (LUT переобучен на bilinear-артефакт). Только после ре-калибровки на реальной камере |

Про яркость главное: **лучший фикс — выключить автоэкспозицию (AEC/AGC) камеры
через SCCB** (фиксированная экспозиция оптическому потоку нужна и так). Тогда ни
high-pass, ни census не нужны, и шумового налога нет. Все яркостно-инвариантные
методы фундаментально усиливают шум.

### Знак результата (важно)

Алгоритм возвращает смещение **текстуры** между кадрами, а не камеры: камера
вправо на `+N` px → текстура влево → `flow_x = -N`. Это оригинальная конвенция
`PX4Flow::compute_flow`, и именно так EKF2 ждёт в DroneCAN-поле `integrated_x`.
**При портировании знак НЕ инвертировать.** `compute_flow_fast(..., invert_sign=True)`
— только для наглядной стрелки на экране (`flow_camera_test.py`), не для железа.

## Валидация на реальных данных

- **Реальная текстура + известная скорость**: улучшения держатся (clean −32%,
  выбросы −78%). Точность до **~3–5%**; Farneback (OpenCV) при этом занижает на
  ~13–17% — наш лёгкий матчер по магнитуде точнее.
- **Снятое видео** (надир, h≈75 см, D≈70 см): 239/239 кадров, боковой дрейф 1.0%.
  Метрически состоятелен — восстановленный FOV совпал на двух клипах. Кажущееся
  расхождение метров было в предположении о FOV (портретное видео ≠ ландшафтный
  паспортный FOV), не в коде.
- **Реальный полёт (rosbag, GPS-denied, 109 м, ЧИСТАЯ камера /cam_usb)**:
  `nav/rosbag_path.py`, `results/nav/bag_trajectory.png`, видео `results/nav/final_run.mp4`.
  Форма совпадает с бортовой оценкой до **1.23 м**, масштаб 0.90 (заякорен
  focal+rangefinder), 2106/2646 валидных, mean q=245. Баг-2 (высоты до 30 м,
  малый охват): resid 1.23 м на 48 м пути (`results/nav/bag2_run.png`).
  Гиро-гейт на 12.5 Гц **не срабатывает ни разу** — «петли на наклоне» были
  артефактом дебаг-топика на 7.8 Гц (`results/nav/bag_gate_compare.png`).
  ⚠️ В баге нет независимого GPS (кастомный OF инжектится как GPS), поэтому это
  кросс-валидация двух OF, не сверка с землёй. Подробно — `docs/FINDINGS.md §3`.

## Стенды безопасности и диапазона

**Де-ротация по гироскопу** (`derotate_test.py`): вращение подделывает трансляцию
(`f·ω·dt`), де-ротация = `flow − f·ω·dt`. Восстанавливает трансляцию, но идёт ПОСЛЕ
измерения → на высоких ω поток вылетает за окно; без пирамиды ломается на ~120 dps,
**с пирамидой до ~200 dps** (синергия). Оси/знак гиро калибруются на железе.

**DroneCAN-выход** (`dronecan_test.py`): в EKF2 уходит `com.hex.equipment.flow.Measurement`
— **интеграл в радианах** `atan2(flow_px, focal)`, не rad/s. Round-trip: скорость
восстанавливается, знак верный. Де-ротация через `rate_gyro_integral` гасит 90 dps
вращения (иначе ложные +1.6 м/с). Стенд ловит ошибку знака гиро-поля.

**Карта диапазона** (`envelope_test.py` → `envelope.png`): потолок захвата `search=4`
(сток H-Flow) = 124°/с, пирамида = 295°/с. Объясняет «на низкой высоте дрон почти не
кренится»: `v/h + ω < ceiling`, на малой h член `v/h` съедает бюджет. Пирамида даёт
2.4× запаса.

**Quality на слепых сценах** (`quality_test.py`): гладкое/тёмное/размытое честно
бракуется (низкий q). Но **повторяющаяся текстура (полосы/плитка) даёт q=255 и врёт**
(aperture, этим страдает и сток H-Flow) — лечит `use_uniqueness`.

## Рекомендации по железу

- **Выключить автоэкспозицию** OV7725 (SCCB, фикс.gain/exposure) — убирает дрейф
  яркости в источнике и «пампинг» потока. Важнее любого яркостно-инвариантного метода.
- **Поднять фреймрейт камеры** (QVGA@120fps) — низкий fps × вращение = поток за
  диапазоном (петли-на-наклоне в баге были от 7.8 fps).
- **Надирный монтаж** + паспортный FOV объектива → метры сразу, без калибровки FOV.
- Калибровать под камеру: `flow_feature_threshold` / `flow_value_threshold` (пороги
  по контрасту), фокус в px для DroneCAN-конверсии.

## C-порт на STM32 — референс ГОТОВ

**`c/px4flow_ref.c`** — портируемая C-реализация полётного набора (median + pyramid
zero-mean coarse + tri-subpix + MAD + boundary), целочисленный тракт (int SAD/ZSAD,
float только субпиксель/агрегация), статические буферы без malloc. NB: MAD в C
захардкожен под golden-приёмку; на железе сделать конфиг-флагом (аудит: над
слаботекстурной землёй MAD выключать).

**Приёмка пройдена:** golden-набор из 20 реальных пар полёта
(`data/test_frames/`, генератор `src/golden.py`) — C совпадает с Python
**точно** (Δflow=0.00000 px, Δq=0 на всех парах). Проверка: `./run_checks.sh`
(собирает gcc и сверяет автоматически).

**Приёмка на плате БЕЗ камеры готова** (`c/stm32/`): golden-набор зашивается во
flash (`src/gen_golden_c.py` → `golden_data.h`), `main_stm32_golden.c` гоняет
20 пар на камне со сверкой 0.05 px и замером тактов DWT->CYCCNT. `__USADA8`
уже встроен в `sad8x8` (`-DPX4FLOW_USE_USADA8`), SIMD-путь доказан бит-в-бит
эмуляцией на ПК, кросс-компиляция M7 проверена. Алгоритм: ≈44 КБ .bss (влезает
в DTCM), данные 360 КБ flash. Инструкция — `c/stm32/README.md`.

Осталось для STM32: подключить DCMI/FIFO-захват, FDCAN (`HAL_FDCAN_*`) + libcanard
(`flow_integral = atan2(flow_px, focal)`, знак НЕ инвертировать — `dronecan_test.py`),
и знаковый тест гиро на столе (чистое вращение ⇒ decoded v≈0). Калибровки — в
`SensorConfig` (`flow_to_velocity.py`). Время мерить `DWT->CYCCNT`.
