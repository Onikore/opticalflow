# PX4Flow algorithm — Python test rig

Прямой Python-порт алгоритма SAD block-matching из
[PX4/PX4-OpticalFlow](https://github.com/PX4/PX4-OpticalFlow)
(`src/px4flow.cpp`), для тестирования на ПК перед портированием
на STM32H743 + ATK-OV7725 MD.

## Структура репозитория

```
src/        код (алгоритм + все стенды), запускать из корня: python3 src/xxx.py
data/       videos/ (снятые .MOV), real_frames/ (текстуры-.npy), test_frames/
results/    рендеры панелей (*.mp4) и графики (envelope.png)
docs/       fallback.md — контекст проекта
```

Пути к данным/выходам в коде считаются от корня репо (`src/_paths.py`),
поэтому скрипты работают из любого каталога.

### Файлы `src/`

Алгоритм:
| Файл | Назначение |
|---|---|
| `px4flow_algo.py` | Эталонная построчная реализация — **1:1 как C++**, референс при портировании на STM32 |
| `px4flow_fast.py` | Векторизованная NumPy-версия — идентичный результат, но быстрая |
| `px4flow_improved.py` | Улучшенный алгоритм. Каждое улучшение — флаг, по умолчанию OFF (бит-в-бит == оригинал). Принятые дефолтом: `use_median`, `use_parabolic` |

Стенды и инструменты:
| Файл | Назначение |
|---|---|
| `benchmark.py` | Честный стенд: субпиксель, шум, дрейф яркости, выброс. RMSE, сравнение версий |
| `test_equivalence.py` | `px4flow_fast` == `px4flow_algo` |
| `synthetic_test.py` | Точность на синтетике с известной скоростью |
| `video_test.py` | Реальная текстура + известная скорость + сверка с Farneback |
| `mov_test.py` | Снятое видео -> метры по геометрии (высота/дистанция) |
| `derotate_test.py` | Де-ротация потока по гироскопу (стенд №1) |
| `dronecan_test.py` | Корректность выхода в EKF2 (единицы/знак/масштаб) (№2) |
| `quality_test.py` | Честность quality на слепых сценах, anti-garbage (№3) |
| `envelope_test.py` | Карта рабочего диапазона -> `results/envelope.png` (№5) |
| `make_panel.py` | Сравнительная панель N алгоритмов -> `results/*.mp4` |
| `flow_camera_test.py` | Интерактивный тест с веб-камерой |

## Запуск

```bash
pip install opencv-python numpy scipy matplotlib

# из корня репозитория:
python3 src/test_equivalence.py          # корректность
python3 src/px4flow_improved.py          # self-check всех улучшений
python3 src/benchmark.py                 # RMSE-сравнение версий
python3 src/quality_test.py              # anti-garbage
python3 src/envelope_test.py             # карта диапазона -> results/

# панель сравнения на своём видео:
python3 src/make_panel.py data/videos/IMG_1252.MOV results/panel.mp4

# живой тест с веб-камерой:
python3 src/flow_camera_test.py --source 0
```

Управление в окне камеры: `q`/`ESC` — выход, `r` — сброс эталонного
кадра, `s` — сохранить текущую пару кадров в `data/test_frames/*.npy`
(пригодится как regression-тест при портировании на STM32 — можно
прогнать те же байты через C-код и свериться с Python-результатом).

## Результаты тестов (уже прогнано)

- **Эквивалентность fast/ref**: 26/26 совпадений, расхождение `0.0`
- **Точность на синтетике**: std ошибки = `0.0000` px (целочисленные
  сдвиги детектируются идеально точно)
- **Производительность чистого Python**: ~4 мс/кадр на блок 64×64,
  ~250 fps эквивалент — это *верхняя* оценка снизу для C-кода на
  STM32H743 (480 МГц), там ожидается 1-3 мс/кадр с CMSIS DSP
  интринсиками (`__SADD8`/`__USAD8` вместо ручных Python-циклов)

## Улучшения алгоритма (реализованы и измерены)

Поверх оригинала добавлены улучшения из плана (`fallback.md`), каждое
включается флагом в `px4flow_improved.compute_flow_improved`. При всех
флагах OFF результат бит-в-бит совпадает с оригиналом (проверено на
800 парах кадров).

Тестировались итеративно через `benchmark.py` (RMSE в пикселях, меньше =
лучше), на 4 сценариях: чистый субпиксель / шум σ=6 / дрейф яркости
(gain 1.12 +8, имитация автоэкспозиции OV7725) / выброс-окклюдер.

### Принято

Надёжный дефолт: **median + parabolic + pyramid + boundary_reject** (scoreboard на реальной
текстуре, `results/scoreboard.png`): clean −32%, шум −46%, **яркость −76%**, выбросы −79%,
без регрессий.

| Улучшение | Флаг | Эффект (RMSE vs оригинал) |
|---|---|---|
| **Медиана вместо среднего** | `use_median` | шум −61%, яркость −35%, выбросы −74%; лёгкий регресс +16% только на идеальном кадре без выбросов |
| **Параболический субпиксель** | `use_parabolic` | clean −48%, яркость −36%, выбросы −30%, шум −22%; **дешевле** оригинального 8-направленного метода (3 готовых значения SAD по оси вместо 8 интерполированных блок-SAD) |
| **Пирамида (coarse-to-fine)** | `use_pyramid` | снимает потолок `search_size=4` (быстрые 5–9 px/кадр −99%). Грубый поиск на ½-разрешении **на zero-mean SAD** (яркостно-устойчив) + уточнение на полном ±2. Раньше ломался на дрейфе яркости (грубый минимум неверный) — починено zero-mean coarse (bright 7.8→0.47, диапазон цел) |
| **Отказ по краю окна** | `use_boundary_reject` | минимум SAD на границе `±search_size` -> блок ненадёжен. На движении за диапазоном роняет quality (255→~140) вместо confident-garbage — без гироскопа. Zero harm на нормальном движении, бонусом bright/шум. Одно сравнение |

### Условно / опционально (флаги, не в дефолте)

| Улучшение | Флаг | Когда включать / почему не дефолт |
|---|---|---|
| Forward-backward consistency | `use_fb_check` | +2× вычислений, и рушит availability именно при дрейфе яркости (8→1 валидных кадра из 40) — а у OV7725 автоэкспозиция дрейфует постоянно. Размен плохой |
| Motion-blur / локализация | `use_peak_quality` | понижает quality по **кривизне корр. пика** (плоский пик = блюр/слабая локализация). Базовый quality уже частично реагирует на блюр (текстура падает → блоков меньше); peak_quality усиливает отклик (на сильном блюре ~вдвое ниже). Уточнение, не must-have. NB: план предлагал «высокий общий SAD» — измерено, что это НЕ работает (симметричный блюр даёт SAD≈0), рабочий сигнал только в кривизне |
| Уникальность минимума (anti-aperture) | `use_uniqueness` | peak-ratio: отбрасывает блок, если второй минимум SAD ≈ лучшему (неоднозначно). **Безопасность:** ловит confident-garbage на повторяющейся текстуре (стенд №3). Цена — availability (шум 40→25, при дрейфе яркости → 0). Включать над плиткой/досками/полем, лучше с фиксированной экспозицией |

### Census transform — условный (флаг `use_census`)

Census заменяет SAD (сумма разностей яркости) на сумму Hamming-расстояний
между локальными бинарными подписями пикселей (для каждого пикселя — 8 бит:
сосед в 3×3 ярче центра на `census_eps` или нет). Подпись зависит только от
**порядка** яркостей, поэтому census **инвариантен к автоэкспозиции**
(монотонное изменение яркости/усиления). `census_eps>0` гасит дрожание бит
у почти-равных пикселей (устойчивость к шуму). Субпиксель — параболический
на cost-объёме (8-направленный интенсивностный метод к битам неприменим).
На STM32 дёшев: сравнения + XOR + popcount по таблице.

Размен на **реальной** текстуре (не строгий апгрейд, поэтому не в дефолте):

| сценарий | SAD median+parab | census | Δ |
|---|---|---|---|
| clean | 0.208 | 0.229 | +10% |
| шум σ=6 | 0.564 | 1.035 | **+83%** |
| **яркость (gain 1.12)** | **1.498** | **0.227** | **−85%** |
| выброс | 0.214 | 0.228 | +6% |

Census уничтожает проблему яркости (на реальной текстуре SAD при смене gain
даёт ошибку ~1.5px!), но платит устойчивостью к шуму.

**Рекомендация — чинить в источнике, а не census'ом:** дрейф яркости у
OV7725 это автоэкспозиция (AEC/AGC). Её надо **выключить через SCCB**
(зафиксировать экспозицию/усиление) — для оптического потока фиксированная
экспозиция нужна и так (AE даёт «пампинг» потока). Тогда проблема яркости
исчезает в источнике, и обычный SAD `median+parabolic` выигрывает везде без
поcensus-налога на шум. `use_census` оставлен как запасной вариант на случай
неустранимых изменений освещения.

## Валидация на реальном видео

Помимо синтетики прогнали на реальных кадрах (`video_test.py`,
`mov_test.py`):

- **Реальная текстура + известная скорость**: улучшения держатся как
  на синтетике (clean −32%, выбросы −78%). Заодно снялся вопрос про
  низкую валидность на синтетике — это был артефакт гладкой
  bilinear-текстуры, на реальных кадрах валидность 32–40/40.
- **Сверка с эталоном Farneback (OpenCV)**: на контролируемой панораме
  с известной истиной наш алгоритм точен до **~3–5%**, а Farneback
  систематически **занижает на ~13–17%** (его сглаживающая регуляризация).
  То есть наш лёгкий блок-матчер по магнитуде точнее эталона.
- **Снятое видео, камера строго вниз, h≈75 см, пройдено ≈70 см**:
  239/239 кадров валидны, quality ~253, боковой дрейф **1.0%**
  (идеальная прямая линия), движение ≤1.1 px/кадр (в диапазоне).

### Ключевой вывод про абсолютные метры

Из двух независимых видео (разные дистанции, одно с наклоном, одно
надир) восстановленный эффективный FOV совпал: **~34–35°**. Один и
тот же FOV при разных дистанциях = алгоритм метрически состоятелен,
пиксели линейно пересчитываются в реальное расстояние.

Кажущееся расхождение «в 2 раза» при первом пересчёте было НЕ в коде:
видео снято **вертикально** (портрет 720×1280), а паспортный
горизонтальный FOV (~65–68°) — для **ландшафта**. У вертикального
видео горизонтальный FOV по короткой стороне примерно вдвое меньше
(~35–41°). С правильным портретным FOV дистанция сходится. На реальном
железе (надирный монтаж + паспортный FOV объектива) этой
неоднозначности нет — тот же код сразу даёт метры.

### Де-ротация по гироскопу (стенд `derotate_test.py`)

У камеры вниз вращение корпуса подделывает трансляцию: наклон с угловой
скоростью ω сдвигает текстуру на `f·ω·dt` px, неотличимо от перемещения.
Поток по сути угловой, поэтому вращательная часть = показанию гиро (не
зависит от высоты), и де-ротация это `flow − f·ω·dt`. Оригинальная плата
PX4Flow тоже несёт гироскоп именно для этого.

Симуляция (трансляция + известное вращение + синтетический гиро,
f≈55px, 30fps):

| ω pitch | сдвиг от вращения | ошибка БЕЗ derot | ошибка С derot (пирамида) |
|---|---|---|---|
| 30 dps | 1.1 px | 0.89 | 0.20 |
| 60 dps | 2.2 px | 2.01 | 0.17 |
| 120 dps | 4.3 px | 4.16 | 0.21 |
| 200 dps | 7.2 px | 7.17 | 0.17 |
| 300 dps | 10.8 px | 7.09 | 3.98 |

Выводы: (1) де-ротация восстанавливает трансляцию из зашумлённого вращением
потока; (2) де-ротация идёт ПОСЛЕ измерения, поэтому полный поток
(трансляция+вращение) надо сначала измерить — а на высоких ω он вылетает за
`search_size`. Без пирамиды де-ротация ломается уже на ~120 dps (сдвиг >4px),
**с пирамидой работает до ~200 dps**, деградирует к 300 (>±10px диапазона).
То есть пирамида и гиро синергичны: вместе дают устойчивость к резким
манёврам. Маппинг осей гиро→поток и знак калибруются на железе.

### Корректность выхода в DroneCAN (стенд `dronecan_test.py`)

В EKF2 уходит `com.hex.equipment.flow.Measurement` — не rad/s, а **интеграл
потока в радианах** (`flow_integral[2] = atan2(flow_px, focal)`) + интервал +
`rate_gyro_integral` для де-ротации. Сквозной round-trip в физике (известная
скорость камеры → пиксельный поток → упаковка → распаковка как EKF →
скорость) на 7 режимах: знак всегда верный, ошибка 0.03–0.14 м/с (растёт с
высотой — `v=rate·h`, пиксельный шум множится на высоту, это физика, не баг).

Отдельно проверена де-ротация через поле сообщения: 90 dps вращения без гиро
декодируется как ложные **+1.6 м/с** (EKF принял бы вращение за полёт), с
`rate_gyro_integral` → ~0. Заодно стенд поймал классическую ошибку: знак
`rate_gyro_integral` должен совпадать со знаком `flow_integral` от вращения —
это калибровка оси/знака гиро на железе, где легко ошибиться (см. запрет
на инверсию знака ниже).

### Карта рабочего диапазона (стенд `envelope_test.py` -> `envelope.png`)

Сенсор меряет не больше `ceiling_px` за кадр. Замерено: `search=4` (сток
H-Flow) = 4.0 px/кадр = **124°/с**, пирамида = 9.5 px/кадр = **295°/с**.
Полный поток `v/h + ω < ceiling_rate`. Отсюда объяснение известного эффекта
«на низкой высоте дрон почти не кренится»: на малой `h` член `v/h` велик даже
при малой скорости и съедает почти весь бюджет — на наклон (ω) не остаётся.
С высотой `v/h` падает, запас растёт. Пирамида поднимает потолок в 2.4×, то
есть на любой высоте разрешает и бо́льшую скорость, и более резкий манёвр до
потери захвата. Карта (v–h и остаточный бюджет наклона по высотам) — в
`envelope.png`.

### Честность quality на «слепых» сценах (стенд `quality_test.py`)

EKF доверяет полю quality, поэтому опасен не отказ, а *уверенно неверный*
поток. Прогон с известным сдвигом по вырожденным сценам:

| сцена | без uniqueness | вердикт |
|---|---|---|
| гладкая / низкий контраст | q=0 | REJECT (честно молчит) |
| тёмная | q=28 | REJECT |
| motion blur | q=48 | REJECT |
| пересвет/блик | q=122, flow верный | GOOD |
| **полосы / мелкая плитка** | **q=255, flow врёт (err 4–6px)** | **DANGER** |

Хорошая новость: гладкое/тёмное/размытое сенсор честно бракует (низкий
quality) — там confident-garbage нет. Плохая: **повторяющаяся текстура
(полосы, плитка, доски, вспаханное поле) даёт q=255 и неверный поток** —
aperture problem, EKF доверится мусору. Штатный PX4Flow/H-Flow этим страдает.

Починка: `use_uniqueness` (переписан на peak-ratio) отбраковывает
неоднозначные блоки -> обе опасные сцены становятся REJECT (q=0). Цена —
availability (см. таблицу флагов). Рекомендация: над самоподобной поверхностью
включать uniqueness + фиксированную экспозицию.

### Замечание про медиану на реальном видео

На снятом видео виден разрыв оригинал vs медиана (при низком
разрешении до +37%): часть блоков «застревает» на нуле из-за смаза от
руки и rolling shutter и тянет среднее вниз. При равномерном движении
над плоским полом все блоки обязаны давать одинаковый поток, поэтому
near-zero блок — это сбой, и медиана правильно его выкидывает. На
большем разрешении (W=96) блоки не сбиваются и разрыв падает до ~8%.
Подтверждает ценность медианы именно на «грязных» реальных кадрах.

## ⚠️ Важно про знак результата

Алгоритм возвращает смещение **текстуры сцены** между кадрами, а не
смещение камеры. Если камера сдвинулась вправо на `+N` px, текстура
в кадре сдвинулась влево — `compute_flow` вернёт `flow_x = -N`.

Это оригинальное соглашение знаков `PX4Flow::compute_flow`, и именно
так PX4 EKF2 это ожидает в DroneCAN-поле `integrated_x`. **При
портировании на STM32 для реальной публикации в DroneCAN знак НЕ
инвертировать.**

`px4flow_fast.compute_flow_fast(..., invert_sign=True)` — опциональный
параметр только для интуитивной визуализации (стрелка "куда летит
дрон" вместо "куда уехала картинка"). В `flow_camera_test.py` он
включён для наглядности экрана.

## План портирования на STM32H743

1. Взять `px4flow_algo.py` как референс по логике (не `_fast.py` —
   та оптимизирована под NumPy broadcasting, не под C)
2. Портировать `compute_diff`, `compute_sad_8x8`, `compute_subpixel`,
   `compute_flow` построчно на C, используя CMSIS DSP интринсики:
   - `__USAD8` / `__USADA8` → встроенные ARM Cortex-M7 инструкции
     (доступны как intrinsics в CMSIS, `core_cmFunc.h`)
   - `__UHADD8` → `__UHADD8` тоже есть как CMSIS intrinsic
3. Кадр 64×64 grayscale = 4 KB — тривиально помещается в RAM (1 MB)
4. Замерить реальное время на железе через `DWT->CYCCNT` (cycle counter)
5. Сверить результат на `data/test_frames/*.npy` парах с Python-эталоном
   побайтово — если совпадает, портирование корректно
6. Подключить вывод в DroneCAN (`com.hex.equipment.flow.Measurement`,
   поля `integrated_x`/`integrated_y` — это `atan2(flow_x, focal_length)`,
   см. `flow_to_dronecan_fields()` в `flow_camera_test.py`)

## Параметры, которые нужно будет откалибровать под реальную камеру

- `FOCAL_LENGTH_PX` в `flow_camera_test.py` — фокусное расстояние
  ATK-OV7725 MD в пикселях для конкретного объектива и разрешения
- `flow_feature_threshold` / `flow_value_threshold` — пороги по
  освещённости/контрасту реальной сцены (тестировались на
  синтетике, для живой камеры может понадобиться подстройка)
